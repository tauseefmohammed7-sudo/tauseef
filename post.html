Automating User Offboarding with PowerShell and Microsoft Graph

JR
Jordan Reyes / Jul 3, 2026 · 8 min read

Every offboarding checklist I've inherited looked the same on paper: disable the account, revoke sessions, transfer files, remove group memberships. In practice, the checklist was a Word document nobody updated, run inconsistently by whoever was on duty that week. The gaps it left — a stale Teams membership here, an orphaned calendar there — are exactly the kind of thing that turns up in a security audit six months later.

This post walks through the script we now run for every offboarding, why it's structured the way it is, and the edge cases that took us two incidents to learn about.

Why manual offboarding fails

Manual processes fail quietly. Nobody notices a missed step until it matters — a departed contractor's OAuth token still working, a shared mailbox still forwarding to a personal address. The fix isn't a better checklist; it's removing the human from the repeatable parts entirely and reserving judgment calls for cases that genuinely need them.

Where this bit us A former employee's calendar delegation outlived their account by three weeks because "remove delegations" was step 11 of 14, and step 11 was the one that got skipped under pressure.

The offboarding script

The script below handles the Microsoft 365 side end to end: disabling sign-in, revoking active sessions, converting the mailbox to shared, and stripping group memberships. It's idempotent — safe to re-run if a step fails partway through.

# Connect once, reuse the session for every step
Connect-MgGraph -Scopes "User.ReadWrite.All", "Group.ReadWrite.All"

function Invoke-Offboarding {
    param(
        [Parameter(Mandatory)][string]$UserPrincipalName,
        [Parameter(Mandatory)][string]$ManagerEmail
    )

    # 1. Block sign-in immediately
    Update-MgUser -UserId $UserPrincipalName -AccountEnabled:$false

    # 2. Revoke all active sessions and refresh tokens
    Revoke-MgUserSignInSession -UserId $UserPrincipalName

    # 3. Strip group and Teams memberships
    Get-MgUserMemberOf -UserId $UserPrincipalName | ForEach-Object {
        Remove-MgGroupMemberByRef -GroupId $_.Id -DirectoryObjectId $UserPrincipalName
    }

    # 4. Grant the manager full mailbox access, then convert to shared
    Set-Mailbox -Identity $UserPrincipalName -Type Shared
    Add-MailboxPermission -Identity $UserPrincipalName -User $ManagerEmail -AccessRights FullAccess
}

Each step is written to be independently safe to retry — disabling an already-disabled account or revoking sessions twice doesn't throw, it just no-ops. That matters more than it sounds like it should, because offboarding rarely happens in one clean pass.

Handling calendar delegates

This was the gap that caused the incident above. Delegated calendar access doesn't get removed by disabling the account or even converting the mailbox — it has to be stripped explicitly.

# Remove calendar delegates before handing the mailbox off
Get-MailboxFolderPermission "$UserPrincipalName:\Calendar" |
    Where-Object { $_.User -notlike "Default" } |
    ForEach-Object {
        Remove-MailboxFolderPermission "$UserPrincipalName:\Calendar" -User $_.User -Confirm:$false
    }
Worth knowing If the departing user organized recurring meetings, consider whether ownership needs to transfer rather than simply removing delegates — otherwise those meetings can be left without an active organizer.

What we still do by hand

Not everything belongs in the script. We intentionally keep three things manual:

  • Data retention decisions — what gets archived versus deleted depends on role and legal hold status, and that's a judgment call.
  • Third-party SaaS access — anything outside Microsoft 365 gets a manual pass, since coverage varies too much to automate safely.
  • Final sign-off — a human confirms the run completed and reviews the audit log before the ticket closes.

Running it safely

Before this runs against a real account, we test it against a disposable test tenant user and check the audit log for anything unexpected. The -WhatIf pattern works for most of these cmdlets and is worth wiring in if you're adapting this for your own environment.

The full version we run in production also posts a summary to a Slack channel and opens a ticket automatically — happy to share that wrapper if there's interest.
